Trace Architecture & Node Reference

A visual guide to every node type in the Trace agentic pipeline builder — how they work, how they connect, and what they do.

System Architecture

Trace is a three-tier system: a React Flow frontend for visual pipeline design, a FastAPI backend that executes the pipeline, and connections to external LLM APIs and MCP servers. Every pipeline is a directed graph that gets topologically sorted and executed node-by-node.

Node Types Reference

Each node type in Trace represents a different function in your agent pipeline. Drag them onto the canvas, connect them, and hit Play.

Deep Dive: Provider Node

FieldTypeDescription
endpointstringLLM API base URL (e.g., https://api.openai.com/v1). Set to "sandbox" for mock mode.
modelstringModel identifier (gpt-4o, o3-mini, claude-sonnet-4, deepseek-v4, mock-gpt-4o)
apiKeystringAuthentication token. Stored in browser localStorage — never sent to the backend unless used in a request.
labelstringDisplay name on the canvas.

Data Flow: Provider → (endpoint, model, apiKey propagate downstream) → Chat makes the actual API call.

Security: API keys stay in the browser. The backend only sees the Authorization header during execution and never persists keys.

Example: Connecting a Provider

Provider (endpoint: "sandbox", model: "mock-gpt-4o")
  │
  ▼
Chat (systemPrompt: "You are a helpful assistant", messages: [{role: "user", content: "Hello!"}])
  │
  ▼
Observer (captures request: {url, headers, body}, response: {choices: [...]})

Deep Dive: Chat Node

FieldTypeDescription
systemPromptstringSystem-level instructions defining the LLM's behavior.
messagesarrayConversation history array of {role, content} objects.
temperaturenumberSampling temperature. 0 = deterministic, 1 = creative. Default 0.7.
labelstringDisplay name on the canvas.

Tool-Calling Flow: When MCP nodes are connected upstream of a Chat node, their tool definitions are automatically collected and included in the LLM request as OpenAI-format tools. The LLM may respond with tool_calls, which the executor handles by:

  1. Parsing the tool name and arguments from the LLM's response
  2. Calling the corresponding MCP tool
  3. Appending the result as a role: "tool" message
  4. Re-calling the LLM with the updated conversation
  5. Repeating until the LLM responds normally (max 10 iterations)

Example: Tool-Calling Loop

User: "What's the weather in NYC?"
  → LLM receives: messages + tools definitions
  → LLM responds: tool_calls(get_weather, args: {location: "NYC"})
  → Executor calls: MCP tool get_weather("NYC")
  → Returns: {"temperature": 72, "conditions": "sunny"}
  → LLM receives: tool result appended as message
  → LLM responds: "The weather in NYC is 72°F and sunny."

Deep Dive: MCP Node

FieldTypeDescription
serverUrlstringURL of the MCP server (e.g., http://localhost:9000)
selectedToolstringCurrently selected tool from discovered tools.
discoveredToolsarrayList of tools fetched from the MCP server, each with name, description, and inputSchema.
toolArgsobjectArguments to pass to the selected tool.

MCP Protocol Flow:

1. Frontend sends: POST /api/mcp/list-tools { serverUrl }
2. Backend attempts: POST {serverUrl}/tools/list  ← JSON-RPC format
   Falls back to:   POST {serverUrl}/list_tools
   Falls back to:   POST {serverUrl}
3. Response: { tools: [{name, description, inputSchema}, ...] }
4. Frontend stores tools in node config
5. During execution: tools are injected into Chat node's LLM request

Deep Dive: Observer Node

FieldTypeDescription
labelstringDisplay name on the canvas.
capturedarrayAccumulated request/response entries from upstream nodes.

What an Observer Shows: For each upstream node, the Observer captures:

  • curl — the exact curl command equivalent of the API call
  • request — URL, headers (redacted), and JSON body
  • response — the full API response (LLM output, tool result, or search results)
  • error — any error that occurred during execution
  • tool_calls — if the step made tool calls, the full history is shown

Deep Dive: Registry Node

FieldTypeDescription
metaobjectMetadata like registry name.

Self-Modifying Pattern: The Registry is the most advanced node. It stores tool definitions in a JSON file on the server. During execution, a Chat node with Registry access can:

  1. Read existing tool definitions from the Registry
  2. Generate new tool code (e.g., "create a calculator tool")
  3. Register the new tool in the Registry via POST /api/registry
  4. The new tool is available in future pipeline runs

Common Pipeline Patterns

1. Simple Chat

Provider → Chat → Observer

The most basic pipeline. Provider holds credentials, Chat makes the LLM call, Observer captures the result. Great for understanding the LLM request/response cycle.

2. Search-Augmented QA (RAG)

Provider → Search → Chat → Observer

Search fetches relevant information from the web, Chat uses it to answer the user's question. The LLM gets fresh context it wasn't trained on.

3. Tool-Calling Agent

Provider → MCP → Chat → Observer

MCP provides tool definitions (e.g., get_weather, calculate, search), Chat includes them in the LLM request. The LLM decides when to call tools.

4. Browser + Summarizer

Provider → Browser → Chat → Observer

Browser fetches a web page's text, Chat summarizes it. Useful for reading news articles or documentation.

5. Self-Modifying Agent

Provider → Registry → Chat → Observer
                └───────── Chat ─────────┘

Registry stores tool definitions. Chat generates new tools and registers them. The agent builds its own capabilities at runtime.

Glossary

TermDefinition
LLMLarge Language Model — an AI model that generates text based on prompts.
MCPModel Context Protocol — a standard for LLMs to discover and call external tools.
RAGRetrieval-Augmented Generation — fetching external info before generating a response.
Tool CallingWhen an LLM requests execution of a function/tool instead of generating text.
Topological SortAlgorithm that orders nodes so each node runs after its dependencies.
System PromptInstructions that define the LLM's behavior and constraints.
JSON-RPCRemote procedure call protocol using JSON — MCP's wire format.
SchemaJSON Schema definition of a tool's input parameters.
CurlCommand-line tool for HTTP requests. Trace generates curl for every pipeline step.
Sandbox ModeBuilt-in mock that simulates API responses without needing real API keys.
StepperStep-through execution mode that pauses at each node for debugging.
A2UIAgent-to-User Interface protocol — renders tool output as rich HTML in an iframe.
